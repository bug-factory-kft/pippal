# PipPal Core v0.2.4 Release Plan

Owner: PO orchestration
Base branch: `main`
Release branch: `release/0.2.4`

## Goal

Make the public Core build easier to start, easier to trust, and smoother
for daily selected-text reading on Windows. The Core line remains useful
without Pro: no accounts, no telemetry, no reading-limit gating, and no
runtime hiding of installed extensions.

## Workstreams

- Daily-use polish: first-run, Settings clarity, tray feedback, reader-panel
  fit, and selected-text reliability.
- Audit and reality checks: README, roadmap, changelog, install docs, test
  counts, live E2E instructions, and release claims must match the repo.
- Compatibility validation: common Windows apps, browsers, PDF surfaces,
  editor/terminal selections, file-open helpers, and context menu behavior.
- Release confidence: unit tests, ruff, live UI E2E when environment allows,
  smoke/install checks, and final PO/QA review before merge.
- Market-backed free value: Core should prove the free promise and support
  Pro conversion by being dependable rather than artificially limited.

## Gates

- `python -m ruff check .`
- `python -m pytest`
- `.\e2e\run-local.ps1` when a live Windows desktop session is available
- `pytest tests/smoke` before public release packaging claims
- README, `CHANGELOG.md`, and `docs/ROADMAP.md` updated together
- No public release/tag activity unless explicitly approved

## Ticket Operating Rules

- One scoped issue per worker-owned change.
- Subagents must comment evidence: commands, result, changed files, and gaps.
- Merge only through a focused PR into this release branch.
- After merging to `main`, pull `main` locally before closing the ticket loop.
- Every large workstream ends with a QA/rethink issue before the next tranche.

## Teaching Note

This uses a release-branch planning pattern. The branch starts with a small
planning artifact so the draft PR has a real coordination surface before code
lands. Alternatives are a wiki-only plan or no release branch until the first
feature. The tradeoff is a tiny planning commit, but every later ticket can
point at the same scope and gates.
